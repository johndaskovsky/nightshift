# AGENTS.md

## Project Overview

This is an AI agent orchestration framework called **Nightshift** — a batch processing system where a manager agent delegates work to dev agents. The framework is distributed as a TypeScript CLI installer (`nightshift init`) that scaffolds agent and command files into target projects, plus Markdown templates that define agent behaviors, commands, specifications, and workflows.

**Runtime:** [OpenCode](https://opencode.ai/) and [Claude Code](https://code.claude.com/) — Nightshift ships subagent and command/skill files for both. The CLI installer accepts `--target=<claude|opencode|both>` (default: auto-detect).
**Package manager:** pnpm.
**Spec framework:** OpenSpec (spec-driven development workflow).

## Build / Lint / Test Commands

### Install dependencies

```bash
pnpm install
```

### Build

```bash
pnpm run build
```

### Tests

```bash
pnpm test                           # init suite + integration suite (auto-detects runtimes)
pnpm test:init                      # fast scaffolder/init suite (no runtime CLI required)
pnpm test:integration               # auto-detect: runs against every installed runtime
pnpm test:integration:opencode      # OpenCode only
pnpm test:integration:claude        # Claude Code only
pnpm test:integration:both          # both runtimes (errors if either CLI is missing)
```

The init suite (`test/init-tests.ts`) exercises scaffolder behavior — directory layout, settings.json merge, CLAUDE.md merge, prose parity — without launching a runtime.

The integration suite (`test/run-tests.ts`) drives shift execution against one or more runtimes. The same fixtures and accuracy checks run under each selected runtime, so we verify behavioral parity. Selection is via `--runtime=<opencode|claude|both>` or `NIGHTSHIFT_TEST_RUNTIMES=...`. With no flag, it auto-detects whichever CLIs are on PATH.

Benchmarks are recorded per runtime (e.g. `nightshift-start.opencode`, `nightshift-start.claude`) so each runtime's perf is tracked independently.

### Shift commands (run inside OpenCode or Claude Code)

| Command | Purpose |
|---------|---------|
| `/nightshift-create <name>` | Scaffold a new shift (manager.md + table.csv) |
| `/nightshift-add-task <name>` | Add a task to an existing shift |
| `/nightshift-update-table <name>` | Bulk modify table data |
| `/nightshift-start <name>` | Begin or resume shift execution |
| `/nightshift-test-task <name>` | Test a single task on one item (no state changes) |
| `/nightshift-archive <name>` | Archive a completed shift |

In Claude Code, all six commands are skills with `disable-model-invocation: true` — Claude does not auto-invoke them, you type `/<name>` explicitly.

### Testing a single task

The closest equivalent to "running a single test" is:

```
/nightshift-test-task <shift-name>
```

This prompts you to select a task and item, invokes the dev agent, and displays results **without modifying `table.csv`**.

## Architecture

### Two-agent system with strict role separation

- **Manager** (`nightshift-manager`): Sole orchestrator. Reads state, delegates to dev via Task tool, writes `manager.md` and task files. Reads `table.csv` for status information but does not write status transitions. Applies step improvements only from successful dev agents. Never executes task steps directly. Can only delegate to `nightshift-dev`.
- **Dev** (`nightshift-dev`): Executor. Follows task steps, self-validates, retries up to 3 attempts, reports step improvement recommendations. Writes its own status to `table.csv` (`done` on success, `failed` on failure) via `flock -x` qsv. Has Playwright access and can create directories. Cannot delegate.

### Item state machine

```
todo → done
    ↘
     failed
```

Status values use snake_case: `todo`, `done`, `failed`.

### Key architectural rules

- Each agent invocation gets **fresh context** (no accumulated state).
- The manager is the **sole writer** of `manager.md` and task files — it does not write status transitions to `table.csv`.
- Dev agents **write their own status** to `table.csv` via `flock -x` qsv commands.
- The dev agent reports step improvement recommendations but never edits task files directly — the manager applies improvements from successful dev agents only.
- Never stop the entire shift for a single item failure (graceful degradation).
- No transient states to recover — items are `todo`, `done`, or `failed`.

## Code Style Guidelines

### File and directory naming

- **kebab-case** for files and directories: `nightshift-dev.md`, `nightshift-commands/`
- **snake_case** for task names: `create_page`, `update_spreadsheet` (task names become CSV column names; hyphens conflict with qsv selectors)
- **Date-prefixed** for archived changes: `2026-02-08-add-nightshift-framework/`
- **Verb-led prefixes** for change IDs: `add-`, `update-`, `remove-`, `refactor-`

### Markdown conventions

- Use `-` for unordered lists (never `*`)
- Use `**bold**` (never `__bold__`)
- Single backticks for identifiers, file names, commands, and status values
- H2 (`##`) for top-level sections, H3 (`###`) for subsections
- One blank line before headings
- Triple-backtick code blocks with language identifier when applicable
- Pipe-delimited tables with alignment row (`|---|---|`)
- No emojis
- No hard line wrapping — paragraphs are single long lines

### YAML frontmatter

All command and agent `.md` files use YAML frontmatter:

```yaml
---
description: <one-line description>
---
```

Agent files additionally include `mode: subagent` and tool/permission blocks.

### CSV conventions

- Header row required
- Status columns named after tasks (snake_case — no hyphens, which conflict with qsv selectors)
- No quoting unless needed

### Specification style

Specs use BDD-style requirement/scenario format with RFC 2119 language:

```markdown
### Requirement: <title>
The system SHALL <normative statement>.

#### Scenario: <scenario name>
- **WHEN** <condition>
- **THEN** <expected behavior>
```

### Error handling patterns

- **Validate-first with early exit**: Check preconditions before proceeding; `STOP` on conflicts.
- **Bounded retry**: Dev agent retries up to 3 total attempts (1 initial + 2 retries).
- **Graceful degradation**: Never halt the batch for a single item failure.
- **Structured error reporting**: Use explicit `FAILED (step N)` or `FAILED (validation)` status.
- **Durable resume**: No transient states — items are `todo`, `done`, or `failed`.
- **Confirmation for destructive ops**: Use `AskUserQuestion` before archiving incomplete shifts.

### Git workflow

- **Branch**: `main` (direct commits, no branching strategy)
- **Commit style**: Imperative mood (e.g., "Add December 2025 status reports")
- **Commit grouping**: Bundle related changes together
- **No force pushing**
